# Design: start.gg API Documentation

**Date:** 2026-05-14

## Goal

Add a permanent, local reference for the start.gg GraphQL API so that agents and developers can consult it directly without hitting the developer portal. The portal has no downloadable schema; the only way to get a machine-readable copy is via introspection.

## Files

```
docs/startgg/
  schema.graphql       Full API schema generated by GraphQL introspection (SDL)
  fetch-schema.sh      Script to re-run introspection and overwrite schema.graphql
  project-notes.md     Curated notes on our 5 operations, quirks, and patterns
```

## schema.graphql

Generated by running an introspection query against `https://api.start.gg/gql/alpha` and converting the JSON result to SDL. Contains every type, field, argument, enum value, and description string the API exposes. This is the machine-readable source of truth for agents.

Committed as a static snapshot. Refresh it before extending the query set by running `fetch-schema.sh`.

## fetch-schema.sh

Shell script located at `docs/startgg/fetch-schema.sh`. Requires `STARTGG_API_KEY` to be set (reads from `.env` if present via a simple `export` or by sourcing `.env`). Uses `npx get-graphql-schema` (no global install needed) to run introspection and write `schema.graphql` in the same directory.

Usage:
```
cd docs/startgg
bash fetch-schema.sh
```

## project-notes.md

Human-readable reference covering:

### API basics
- **Endpoint:** `https://api.start.gg/gql/alpha` (stable production path despite the name)
- **Auth:** `Authorization: Bearer <STARTGG_API_KEY>` header; no OAuth needed for read queries
- **Rate limits:** 80 requests per 60 seconds; 1000 nested objects per request (complexity cap)

### Our 5 operations
Each documented with: purpose, the query string as used in code, important fields and why they're selected, and pagination approach.

1. `GAME_SEARCH_QUERY` — find a videogame ID by name
2. `USER_BY_SLUG_QUERY` — resolve a start.gg user slug to an internal user ID
3. `TOURNAMENTS_BY_USER_QUERY` — paginate all tournaments a user attended for a given game; includes nested events → phases → phaseGroups
4. `EVENT_ENTRANTS_QUERY` — paginate all entrants in an event (with participant/user identity)
5. `EVENT_SETS_QUERY` — paginate all completed sets in an event (with slots, entrants, DQ flags, scores)

### Known quirks
- **`ActivityState` type inconsistency:** The API documents `ActivityState` as an enum, but returns it as a string for `Event` and `Phase` fields, and as an integer for `Tournament`, `PhaseGroup`, and `Set` fields. Deserialisation must handle both (`#[serde(untagged)]` or per-field `deserialize_with`).
- **Pagination:** All collection queries use `page`/`perPage` variables. There is no cursor-based pagination. `PageInfo.totalPages` is the correct stopping condition; `PageInfo.total` may be null on some endpoints.
- **DQ detection:** A set where one slot's `standing.stats.score.value` is `-1` is a DQ, not a real result. Filter these before computing upset factor.

## Testing

No new tests. The schema file and notes are reference material; `fetch-schema.sh` is a developer tool, not part of the build or CI pipeline.

## Out of scope

- Generating Rust types from the schema (would require a codegen tool and ongoing maintenance; the current hand-written types in `crates/common/src/startgg/queries.rs` are sufficient)
- Scraping the developer portal narrative docs (high effort, low additional signal over the SDL)
- Automating schema refresh in CI
